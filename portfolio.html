IKIWISI: Visual Debugging for Large Multi-Modal Model Inconsistencies

🏆 Best Paper Honorable Mention · DIS'25 ACM DL GitHub
PyTorchDash Plotlynumpy RStudioSPSS

Problem

Large multimodal models (LMMs) like GPT-4V can process text, images, and video — but unlike humans, their outputs often lack common sense and can be inconsistent across modalities. In multi-label video object recognition, a model might identify an object in one frame but miss it in the next, even when frames are nearly identical.

Automated metrics like F1 or average precision cannot capture such inconsistencies, making human evaluation essential. But there was no simple tool that let non-experts assess whether a model was actually performing reliably.

Solution

IKIWISI (I-Know-It-When-I-See-It) is a lightweight, intuitive tool that visualizes LMM prediction inconsistencies across video frames via binary heatmaps:

  • The user selects objects of interest and a video from dropdowns.
  • Up to 16 keyframes are displayed in an image container for inspection.
  • A binary heatmap represents model predictions — each row is an object, each column a video frame.
  • High-level patterns in the heatmap let users quickly judge model reliability without any ML expertise.

Outcomes

  • In a user study with 15 participants, the heatmap visualization enabled identification of subtle inconsistencies — models flipping answers across turns or hallucinating details.
  • Participants' ratings of model performance correlated strongly with actual F1 scores.
  • Users inspected only a tiny fraction of frames to judge a model, thanks to high-level heatmap patterns.
  • Even non-technical users could identify inconsistency patterns and contribute to model assessment workflows.

Impact

IKIWISI is among the first tools to enable intuitive inconsistency diagnosis across LMM outputs without requiring code or model access. It supports faster and more inclusive evaluation workflows for researchers, QA teams, and designers, and can be integrated into model audit pipelines and prompt refinement tools.

Wheeler: Multi-Wheel Input for Non-Visual Navigation

🏆 Best Paper Honorable Mention · UIST'24 Paper Promo Presentation
Hardware Blind Users Input Device GUI Navigation Arduino Pro Micro C# C++

Problem

Blind users often face major challenges when navigating modern graphical user interfaces, especially when interface elements are deeply nested, unlabeled, or visually grouped without accessible structure.

Traditional screen readers and keyboard shortcuts are often linear and slow, making it difficult to work efficiently in complex software or perform tasks such as remote technical support. These barriers can increase reliance on sighted assistance and reduce digital autonomy.

Solution

Wheeler is a mouse-shaped device with three independently rotatable wheels, where each wheel controls a different axis or level of navigation.

  • H-Nav Mode: supports hierarchical navigation through app structure, such as window → panel → element.
  • 2D-Nav Mode: supports spatial navigation, where two wheels control horizontal and vertical cursor movement and the third wheel controls cursor speed.
  • The device uses rotary encoders along with haptic and audio feedback.
  • It was iteratively co-designed with blind users through participatory design sessions.

Evaluation

  • Two controlled lab studies with 12 blind participants.
  • A 6-month diary study with a blind co-author using Wheeler in real-world scenarios.

Outcomes

  • Reduced navigation time by up to 40% in hierarchical tasks compared to screen reader + keyboard interaction.
  • Helped users access unlabeled or hidden UI elements such as sliders, tooltips, and aria-hidden items.
  • Showed real-world utility in productivity tasks, web browsing, and remote collaboration.
  • Users reported improved agency, lower frustration, and quick learnability.

Impact

Wheeler introduces a new rotary, parallel input paradigm for non-visual interaction. It enables faster and more structured access to complex application hierarchies, supports interaction with otherwise inaccessible interface components, and offers a low-cost augmentative tool that works alongside screen readers.

SpaceXMag: Automatic Space Compaction for Low-Vision Accessibility

IMWUT'23 / UbiComp Paper Presentation GitHub
Low Vision Mobile UI Optimization GEKKO RICO-SCA Accessibility

Problem

Low-vision users often rely on screen magnifiers to interact with smartphone interfaces. But conventional magnifiers enlarge both useful content and redundant whitespace, which forces excessive panning and frequently causes loss of context.

Because these tools do not reason about layout structure, they can make everyday mobile interaction slower, more effortful, and more cognitively demanding.

Solution

SpaceXMag is an optimization-based system that automatically compacts whitespace in smartphone UIs while preserving structural relationships between interface elements.

  • Operates on DOM-based UI metadata, not raw pixels.
  • Uses nonlinear optimization with GEKKO/IPOPT to restructure layouts under element constraints.
  • Compacts screens through a bottom-up hierarchical traversal for local flexibility and global consistency.
  • Supports fullscreen, window, and fisheye magnifier modes.
  • Includes a rendering backend to reconstruct the modified layouts visually.

Evaluation

  • User study with 11 low-vision participants.
  • Large-scale testing on 16,500+ screens from the RICO-SCA dataset.

Outcomes

  • 28.13% faster in overview tasks than baseline magnifiers.
  • 42.89% faster in target acquisition tasks.
  • Users reported better screen awareness, lower zooming effort, and higher satisfaction, especially in window mode.
  • Saved an average of 47.17% whitespace across 16.5K+ real app screens.
  • Median runtime was 1.44 seconds per screen, up to 60× faster than prior solutions like SUPPLE.

Impact

SpaceXMag demonstrates that DOM-aware, optimization-driven layout restructuring can make magnification substantially more usable in real-world mobile interfaces. It points toward smarter adaptive accessibility tools that preserve context instead of simply enlarging pixels.

TeachPilot: Instructor-Guided LLM Support for Reasoning-First Learning

Classroom LLM System GitHub
Streamlit Ollama Education Prompt Control Supabase Assignments

Problem

In classroom settings, general-purpose LLMs can easily over-help, provide direct solutions too quickly, or ignore the structure and expectations of a specific assignment. This makes it hard for instructors to use AI support without undermining students’ own reasoning process.

Instructors also need a practical way to adapt AI behavior across assignments, manage student access, and review interactions in a structured classroom environment.

Solution

TeachPilot is a teacher-guided LLM assistant that changes behavior per assignment and applies instructor-defined guardrails before sending requests to the model.

  • Uses a two-part prompting structure: a base system prompt plus an assignment-specific prompt.
  • Supports assignment-scoped conversations, so each chat is tied to the active coursework context.
  • Allows text, image, and file uploads in chat when using compatible vision-language models.
  • Provides admin controls for assignments, prompts, users, and conversation browsing.
  • Provides a student-facing chat experience with copy-message and copy-conversation utilities.

System Design

  • Built with Streamlit and Ollama Cloud.
  • Persists chats and metadata in Supabase Postgres, with optional Supabase Storage for uploads.
  • Supports role-based use: admin manages assignments/prompts/users, while students use chat only.
  • Processes uploaded images and files before composing the final model request.

Outcomes

  • Creates a more controlled classroom AI workflow centered on reasoning-first support instead of unrestricted answer delivery.
  • Gives instructors direct control over assignment framing, help constraints, and conversation review.
  • Makes it feasible to deploy course-specific AI support with persistent history, transcript access, and manageable administration.

Impact

TeachPilot shows how classroom AI can be structured as an instructor-shaped system rather than a generic chatbot. By combining assignment-aware prompting, role-based controls, and persistent conversation management, it supports more accountable and pedagogically aligned AI use in education.

Identifying Crucial Objects in Blind and Low-Vision Individuals’ Navigation

ASSETS'24 Paper GitHub
Blind Users Low Vision Object Taxonomy Benchmarking Vision Models Accessibility

Problem

Blind and low-vision (BLV) individuals face major challenges when navigating urban and indoor environments, but existing computer vision datasets often miss many objects that are critical for safe movement.

Common datasets such as ImageNet and MS-COCO do not adequately represent hazards and navigation-relevant objects like overhanging branches, sidewalk pits, or bus stops, limiting the usefulness of current vision systems for accessibility.

Solution

This project created an accessibility-centered object taxonomy and benchmark specifically for BLV navigation.

  • Analyzed 21 publicly available navigation videos involving BLV individuals.
  • Identified 80+ candidate objects that affected navigation through interaction, guidance, or hazard potential.
  • Refined the list through a 6-person focus group with blind, low-vision, and sighted accessibility experts.
  • Produced a final set of 90 objects across 15 semantic categories.
  • Labeled 31 video segments with object presence and absence annotations.
  • Benchmarked seven vision models including detection, segmentation, and open-vocabulary systems.

Outcomes

  • Modern vision models performed poorly on many accessibility-critical objects.
  • Detection and segmentation models typically recognized only 12–15 objects.
  • Open-vocabulary models performed better, but still had incomplete coverage.
  • BLIP achieved the best reported result with an F1 score of 0.48, yet still missed important safety-relevant objects such as barrier stumps and tree branches.

Impact

This work highlights a major gap between state-of-the-art vision systems and the real object awareness needed for safe BLV navigation. By releasing a benchmark dataset and object taxonomy grounded in accessibility needs, it lays the foundation for more inclusive navigation aids and future accessibility-focused vision models.

Quantifying Perceived Accessibility in Non-visual Interaction

CHI'23 Paper GitHub
C# Python NVDA Accessibility APIs Probabilistic Modeling Desktop UI

Problem

Blind screen reader users rely on keyboard navigation to interact with desktop applications, but navigating menus, toolbars, dialogs, and nested interface structures is often slow, inconsistent, and frustrating.

Existing accessibility evaluation methods usually focus on compliance checks or labor-intensive user studies. They do not provide a scalable way to estimate perceived accessibility — how accessible an application actually feels to blind users during real non-visual interaction.

Solution

This project introduces a probabilistic interaction framework for estimating perceived accessibility from the structure of desktop interfaces and keystroke-based navigation behavior.

  • Built from findings of a formative study with 11 blind participants.
  • Uses UI hierarchies extracted through accessibility APIs, without needing pixel data or screen recordings.
  • Models navigation effort through a probabilistic interaction model and keystroke cost analysis.
  • Defines three task-agnostic metrics: Complexity, Coverage, and Reachability.
  • Benchmarks accessibility across 11 commonly used Windows applications.

Metrics

  • Complexity: expected keystrokes needed to reach a target UI element.
  • Coverage: percentage of the interface reachable within a fixed keystroke budget.
  • Reachability: fraction of the interface that can be accessed efficiently using available shortcuts and navigation structure.

Outcomes

  • The metrics revealed clear perceived-accessibility differences across applications.
  • Notepad showed low complexity (1.63), while Microsoft Word showed substantially higher complexity (3.51).
  • Participants described the metrics as intuitive and reflective of their actual navigation effort.
  • Applications with high complexity and low coverage were consistently experienced as more frustrating and difficult to use.

Impact

This work provides an automated, user-centered way to estimate perceived accessibility for non-visual desktop interaction. It helps developers and auditors compare applications, identify concrete navigation bottlenecks, and move accessibility evaluation toward scalable, interface-aware assessment rather than compliance alone.

OpenAccAPI: An Open Source Accessibility API for 3D and VR Apps

Under Review GitHub
Unity XR Accessibility API Screen Readers SteamVR Meta Quest Link

Problem

Blind and low-vision users face major barriers in Unity-based 3D and VR applications because these apps often lack screen reader support, accessible UI components, and effective keyboard navigation.

Existing accessibility plugins are fragmented, visually oriented, and usually must be built into the game during development, which makes them hard to retrofit and inadequate for non-visual interaction in existing immersive applications.

Solution

OpenAccAPI is an extensible accessibility API for 3D and VR applications that exposes UI and scene elements to assistive technologies and supports retrofitting accessibility into existing Unity apps.

  • Provides access to scene and interface elements for assistive tools.
  • Enables features such as text-to-speech, UI text highlighting, non-visual navigation, and cursor orientation feedback.
  • Includes a reference implementation for Unity.
  • Supports plugin injection into existing Unity games in three clicks.

Evaluation

  • Tested on 15 popular Unity games on Steam.
  • Most features worked smoothly in 12 of the 15 games.
  • Small-scale studies with blindfolded users suggested better spatial awareness and navigation capability in 3D and VR settings.

Outcomes

  • Showed that accessibility features can be retrofitted into existing Unity applications rather than only added during development.
  • Demonstrated generalizability across a diverse set of real Unity games.
  • Reduced the effort needed for developers to add accessibility support in immersive environments.

Impact

OpenAccAPI introduces a comprehensive accessibility API for 3D and VR apps aimed at non-visual interaction. It creates a practical path toward more accessible immersive software and lays groundwork for future research in accessible XR design and agent-based non-visual navigation.
